# 智擎模型服务平台定制化建设汇报

## 1. 汇报摘要

智擎模型服务平台基于 `new-api` 构建。在保留上游统一模型网关、用户与密钥管理、计量计费、渠道管理等基础能力的前提下，本次工作围绕本地多节点模型服务的可观测性、运营分析和内部审计进行了定制化建设。

本轮定制的核心目标是把“能调用模型”的网关，升级为“可看见、可分析、可追溯、可运维”的模型服务平台：

- 面向普通用户，提供模型服务健康状态和个人/密钥/模型维度的调用分析。
- 面向管理员，提供渠道、网关、vLLM 引擎和物理主机的统一运行视图。
- 面向管理者，提供用户规模、渠道规模、调用量、Token 用量、成功率、时延等可直接解读的运营指标。
- 面向内部治理，支持按完整对话轮次留存请求参数、输入、输出与推理内容，并使用独立存储避免影响网关主库。

截至本次梳理，已提交到当前主分支的定制提交为 21 个，较合入上游基线增加约 5,500 行代码；当前工作区另有 vLLM 原生指标和对话审计等已完成开发、待纳入正式发布分支的能力。

## 2. 建设范围与分层

| 层级 | 建设重点 | 面向对象 |
| --- | --- | --- |
| 服务层 | 渠道健康检查、vLLM 原生指标采集、对话审计 | 平台运行与模型服务 |
| 数据层 | 健康检查历史、主机资源采样、vLLM 指标聚合、ClickHouse 对话审计库 | 运维与长期分析 |
| 应用层 | 服务监控、运维大屏、资源监控、模型调用分析、审计查询 | 用户、管理员、管理者 |
| 体验层 | 中文数字与时间格式、人民币定价展示联动、登录后深链返回 | 全体使用者 |

## 3. 相对上游的主要优化

### 3.1 服务监控：从外部状态页转为平台原生监控

**上游基础能力**：提供渠道配置、通用日志和基础仪表盘，但没有针对本地模型渠道的原生、按渠道展示的连续健康时间线。

**定制实现**：

- 对所有已启用渠道按分钟执行原生健康检查，禁用或删除渠道会自动从监控视图移除。
- 保存渠道可用性和响应时间历史，服务监控页展示最近 24 小时的状态时间线、成功率和最近响应时延。
- vLLM 渠道额外呈现输出 Token/s、运行中请求和排队请求，使服务状态不只停留在“通或不通”。
- 服务监控向普通用户开放，管理员无需维护独立的 Uptime Kuma 状态页。

**业务价值**：用户可以自行判断模型是否可用，运维人员可以快速定位是渠道异常、响应变慢还是模型实例排队，减少人工问询和多系统切换。

### 3.2 运维大屏：建立面向管理者的运行总览

**定制实现**：

- 新增独立的“运维大屏”，按最近 24 小时汇总网关调用、Token 用量、成功率、平均调用耗时、用户与渠道规模。
- 以趋势图展示成功/失败请求、Token、成功率和调用时延；空闲时段的成功率按 100% 表示，避免“无调用”被误读为“调用失败”。
- 接入 vLLM 运行请求、排队请求、输出吞吐、解码速度、首 Token 时延、前缀缓存命中率和 KV Cache 使用情况。
- 支持管理员导出监控数据，便于形成日常运行报告。
- 全部趋势数据统一按北京时间展示，并修复了国际化语言标识导致的图表错误。

**业务价值**：将用户、模型、渠道、网关和推理引擎的运行信息集中到同一页面，便于领导快速判断平台规模、服务质量和资源压力。

### 3.3 资源监控：将多台物理机纳入统一视图

**定制实现**：

- 管理员可配置需要监控的主机地址、SSH 端口、用户名和私钥。
- 私钥以 AES-GCM 加密后存储，不在接口响应和前端页面中返回明文。
- 每 30 秒采集 CPU 使用率、内存使用量、GPU 利用率、显存、温度和功耗；保留最近 7 天的采样历史。
- 资源监控页面按主机展示概览、CPU/内存趋势、每张 GPU 的实时状态，并关联该主机上的渠道和 vLLM 实例。
- 仅管理员可访问主机配置和资源监控页面。

**业务价值**：平台从“只看 API 是否成功”扩展为“同时看模型服务和承载资源”，可提前识别显存紧张、GPU 过载和节点异常。

### 3.4 vLLM 原生指标：用引擎指标替代日志估算

**交付状态：已完成开发，待随当前工作区变更纳入发布。**

**定制实现**：

- 面向已启用的本地 vLLM 渠道，直接抓取实例 `/metrics`，而非仅由网关日志反推吞吐与并发。
- 每 15 秒采集运行中请求、排队请求、生成 Token 累计值、TTFT、解码间隔、前缀缓存和 KV Cache 指标。
- 计算输出 Token/s、解码 Token/s、平均首 Token 时延、缓存命中率与最大 KV Cache 并发能力。
- 原始样本保留 7 天，按 5 分钟窗口聚合的数据保留 90 天，兼顾实时查看与趋势分析。
- 指标与渠道 ID 绑定，并进一步关联到服务监控、运维大屏和资源监控的主机视图。

**业务价值**：对本地 vLLM 场景，吞吐、并发和排队指标以引擎真实运行状态为准，可为扩容、模型调参和性能评估提供可信依据。

### 3.5 模型调用分析：细化到用户、密钥和模型

**定制实现**：

- 在数据看板中新增模型调用分析。
- 基于消费日志按用户、API 密钥、模型和时间聚合，分别展示输入 Token、输出 Token、总 Token 和请求次数。
- 提供用户/模型 Token 排行、Token 趋势和可筛选的明细表，支持时间范围、时间粒度、用户、密钥和模型维度筛选。

**业务价值**：可准确回答“谁在使用、使用了哪个模型、使用量是多少、输入输出结构如何”，为配额管理、成本核算和服务推广提供数据基础。

### 3.6 对话审计：以完整对话轮次留存内部调用记录

**交付状态：已完成开发，待随当前工作区变更纳入发布。**

**定制实现**：

- 对 Chat Completions、Completions、Responses 和 Gemini 等对话请求进行轮次级审计。
- 记录用户、密钥、模型、渠道、请求 IP、请求路径、请求参数、输入消息、最终输出、推理内容、状态码、结束原因、首响应时延和总耗时。
- 支持通过请求中的 `conversation_id`、元数据或平台请求头关联连续对话；未提供时以请求 ID 兜底。
- 对流式响应在内存中累积已生成内容，客户端中断时仍可保留已经生成的前缀，而不保存零散的流式分片。
- 对话内容写入独立 ClickHouse 库，采用异步队列和批量落库；审计库不可用时自动降级，不阻断模型转发。
- 审计查询和详情接口仅管理员可访问。

**业务价值**：满足内部服务的质量复盘、问题定位和数据清洗需求，同时避免把大文本审计数据压入网关主数据库。

### 3.7 本地化与使用体验改进

- 修复中文界面使用 `zhCN` 作为语言标识导致浏览器 `Intl` 格式化异常的问题，统一使用标准语言标识。
- 登录成功后保留原始访问路径，用户可回到此前打开的深层页面。
- 在人民币展示模式下，模型定价编辑器按汇率进行显示值与美元存储值的双向换算，避免“前台显示人民币、编辑仍只接受美元”的体验断裂。该修复目前位于独立发布分支，建议与主分支定制能力一并合并发布。

## 4. 数据与安全设计

| 数据类型 | 存储位置 | 用途 | 保留策略 |
| --- | --- | --- | --- |
| 网关消费与调用日志 | 平台日志库 | 用量分析、调用追踪 | 沿用平台日志策略 |
| 渠道健康检查 | 平台主库 | 服务时间线、成功率、响应时延 | 7 天 |
| 主机资源采样 | 平台主库 | CPU、内存、GPU 历史趋势 | 7 天 |
| vLLM 原始指标 | 平台主库 | 实时吞吐、并发与排队分析 | 7 天 |
| vLLM 聚合指标 | 平台主库 | 长周期性能趋势 | 90 天 |
| 对话审计内容 | 独立 ClickHouse | 完整轮次审计与离线清洗 | 不自动清理，按管理策略长期保留 |

关键安全措施：

- 主机 SSH 私钥使用 AES-GCM 加密存储，接口只返回“是否已配置”。
- 资源监控、对话审计和运维大屏均要求管理员角色。
- 对话审计采用可选配置：未设置 ClickHouse 连接时功能自动关闭，不影响网关转发。
- 渠道、主机和 vLLM 指标仅采集已启用对象，减少无效采样与错误告警。

## 5. 可用于汇报的成效表述

1. 完成模型服务平台从 API 网关向运维运营一体化平台的升级，覆盖服务健康、模型性能、主机资源和调用行为四类核心数据。
2. 建立面向普通用户、管理员和管理者的分层视图，既降低用户排障成本，也提升管理层对平台规模和运行质量的可见性。
3. 将本地 vLLM 引擎的吞吐、并发、排队、时延和缓存指标直接纳入平台监控，为多节点部署的容量规划提供数据支撑。
4. 将主机 CPU、内存和 GPU 多卡状态纳入统一管理，形成“网关 - 渠道 - 引擎 - 物理机”的完整观测链路。
5. 建立独立 ClickHouse 对话审计能力，在不影响模型响应链路的前提下保留完整对话轮次，支持后续质量分析与数据治理。

## 6. 建议下一阶段工作

1. **指标准确性**：把运维大屏中的成功/失败请求由成功率反推改为后端精确计数，确保所有图表均可审计、无歧义。
2. **告警闭环**：增加渠道不可用、错误率突增、排队请求持续增长、GPU 显存或温度超阈值的通知与确认机制。
3. **容量规划**：增加 7 天/30 天趋势、峰值利用率、资源余量和高负载持续时间，支撑节点扩容决策。
4. **审计治理**：补充敏感字段脱敏、审计导出权限和访问留痕，明确长期保存数据的使用边界。
5. **发布治理**：将自定义模块保持在独立的 controller/service/feature 目录，优先以增量接口扩展上游，降低后续同步官方更新的合并冲突。

## 7. 术语表

| 术语 | 含义 |
| --- | --- |
| 渠道 | 平台中配置的一个上游模型服务实例或接入端点。 |
| 服务监控 | 以渠道为单位展示可用性、响应时延和 24 小时状态时间线。 |
| 运维大屏 | 面向管理者汇总网关调用、服务质量、用户渠道规模和 vLLM 运行指标的页面。 |
| 资源监控 | 通过 SSH 采集并展示承载模型服务的物理机 CPU、内存和 GPU 指标。 |
| 输出 Token/s | 单位时间内模型生成的 Token 数，反映生成吞吐能力。 |
| 解码 Token/s | 模型解码阶段的 Token 生成速度，反映单请求生成速度。 |
| TTFT | Time To First Token，收到请求到返回首个 Token 的平均时延。 |
| KV Cache | 推理引擎保存上下文计算结果的缓存，影响并发能力和重复上下文的性能。 |
| 对话审计 | 按完整请求/响应轮次保存内部模型调用记录的能力。 |

## 8. 发布说明

本报告将功能按代码状态分为两类：

- **已纳入当前主分支**：服务监控、运维大屏、资源监控、模型调用分析、登录深链和中文格式化修复等。
- **已完成开发、待纳入正式发布分支**：vLLM 原生 `/metrics` 采集、ClickHouse 对话审计及其管理页面。

正式发布前应在测试环境完成配置校验、权限验证、指标对照和数据迁移验证，再发布至生产环境。
